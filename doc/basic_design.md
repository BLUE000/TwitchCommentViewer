# Twitchコメント解析ツール 基本設計書

## 1. システムアーキテクチャ方針

本システムでは、Qtの特性を活かし、UI層と内部ロジック、各連携機能を完全に分離した「メッセージパッシング方式」を採用する。

### 1.1 通信プロトコル（QMetaObject::invokeMethodの活用）
* モジュール間やスレッド間の通信において、シグナル＆スロットの静的な `connect` は極力使用しない。
* 代わりに、`QMetaObject::invokeMethod(target, "methodName", Qt::QueuedConnection, ...)` を利用する。
* これにより、関数の呼び出し要求が自動的に「イベント」に変換され、対象スレッドのイベントキューに `postEvent` されるため、非常に疎結合かつスレッドセーフなアーキテクチャが実現する。

### 1.2 スレッド分離モデル（Thread Isolation Model）
* メインスレッド（UIスレッド）の応答性を極限まで高めるため、以下のモジュールはそれぞれ独立したバックグラウンドスレッド（`QThread`）上で稼働させる。
  - `TwitchEventCollectorImpl` (WebSocket通信およびAPIクライアント)
  - `CommentAnalyzer` (形態素解析・感情分析)
  - `DatabaseManager` (暗号化SQLiteアクセス)
  - `BouyomiIntegrationImpl` / `VoiceVoxIntegrationImpl` (音声読み上げ連携用モジュール)
  - `ObsHttpServer` (OBS連携用HTTPサーバー)
* UIスレッドで稼働する `AppController` が仲介役となり、各ワーカースレッドに対して `Qt::QueuedConnection` を用いて非同期で指令をディスパッチし、結果をUIに伝える。これにより、コメントの大量流入時やAPIリクエスト待ちが発生した場合でも、画面のフリーズを完全に防ぐ。

## 2. モジュール構成とデータフロー

### 2.1 UI層 (Presentation Layer)
* **構成**: Qtの `.ui` ファイルと対応するUIクラス。
* **役割**:
  * ユーザー操作を起点とし、`on_pushButton_clicked()` 等のイベントハンドラから、コントローラ層へ `invokeMethod` で処理を依頼する（行き）。
  * 内部処理モジュールから `invokeMethod` によって投げられた「UI更新イベント」を受け取り、画面を描画する（帰り）。

### 2.2 コントローラ層 (Core Application Layer)
* **役割**:
  * 各モジュールのインスタンス生成と管理。
  * 各モジュールをバックグラウンドスレッド（`QThread`）に配置し、適切な通信ルートを確立する。

### 2.3 モジュール層 (External & Internal Services Layer)
各機能は完全に独立したモジュールとして実装する。

#### 外部インターフェースモジュール
1. **Twitch受信モジュール (`TwitchEventCollector`)**
   - EventSub(WebSocket)からコメントやイベントを受信し、システムに通知する。
2. **Twitch送信・操作モジュール (`TwitchActionSender` / `TwitchEventCollectorImpl`)**
   - コントローラからの要求を受け取り、Twitch API(HTTP)を実行してコメント送信、モデレーション操作、ピン留め操作、および**アナウンス送信**を行う。
3. **OBS連携モジュール (`ObsIntegration` / `ObsHttpServer`)**
   - HTTPサーバー経由でOBS向けのオーバーレイ表示HTMLを提供し、チャット表示等の動的な演出を行う。
4. **読み上げ連携モジュール (`ITtsIntegration` / `BouyomiIntegrationImpl` / `VoiceVoxIntegrationImpl`)**
   - 棒読みちゃん(TCP)やVOICEVOX(HTTP API & Win32 PlaySound)へのテキスト送信を担当。
   - **TTS読み上げ除外フィルター**: 設定管理モジュールと連携し、ボットユーザーや特定の除外ユーザーリストに登録されたチャットは送信前に自動でスキップする。

#### 内部処理モジュール
5. **データ管理モジュール (`DatabaseManager`)**
   - `TransCipher-Dist` ライブラリを用いたデータの難読化と、SQLiteへの書き込み・読み込み。
6. **設定管理モジュール (`ConfigManager`)**
   - 安全なOAuthトークン管理のために、Windows環境ではOSの **DPAPI (Data Protection API)** を使用して資格情報を暗号化。非Windows環境では `TransCipher-Dist` にフォールバックする。
   - アナウンス送信用に新しいOAuthスコープ `moderator:manage:announcements` を含めて認可フローを起動する。
7. **コメント解析モジュール (`CommentAnalyzer`)**
   - 受信したコメントを解析（形態素解析、感情判定、スパム判定等）し、結果をシステムに非同期で返す。

## 2.4 視聴者操作・モデレーションの設計

* **ホバー・選択時表示UI**:
  - `QListWidget` の各行にカスタムウィジェットを配置。
  - マウスホバー（enterEvent / leaveEvent）またはクリックによるアイテム選択をトリガーとし、右端の操作ボタン群を表示・非表示制御。
* **トグル状態管理**:
  - `GET /chat/chatters` APIで取得したバッジデータを初期状態としてボタンのトグル（オン/オフ）を設定。
  - VIP・モデレータ操作は、トグル状態に応じて「追加API」と「削除API」を切り替えて呼び出す。
* **確認と安全性の設計**:
  - BAN、モデレータ変更、タイムアウト実行時には、実行前に `QMessageBox` による確認ダイアログを表示して誤操作を防ぐ。
  - タイムアウトは数値指定ダイアログにより任意の時間を指定。
* **シャウトアウトのクールタイム管理**:
  - 実行時にタイマーを起動し、2分間のクールタイムをカウントダウン。残り時間をウィンドウのタイトルバーに表示し、ボタンを無効化。
* **視聴者リストの複数カテゴリ分類**:
  - 取得した視聴者は、ストリーマー、モデレーター、VIP、アーティスト（手動/自動判定）、チャットボット（設定で指定されたBot）、一般ユーザーに分類し、それぞれアコーディオン（折りたたみ可能なウィジェット）に表示する。
  - 各アコーディオンは排他的ではなく、**重複分類**をサポートする（例: Nightbotは「モデレーター」と「チャットボット」の両方に重複表示される）。

## 2.5 コメントピン留め・アンピン機能の設計

* **UI連携**:
  - コメント表示テーブル (`tableViewComments`) を右クリックした際に表示されるコンテキストメニューから起動。
  - 指定時間（30秒、1分、5分、15分、30分、配信終了まで）を選択可能。
  - ピン留め中である場合は、「ピン留めを解除」アクションも動的に表示。
* **非同期API送信**:
  - UIからコントローラへ非同期メッセージパッシング。
  - `TwitchEventCollectorImpl` から Twitch APIの `POST /helix/chat/pins` および `DELETE /helix/chat/pins` を呼び出してチャット最上部にピン留めする。

## 2.6 コメントグラフの自動目盛り調整・ウィンドウサイズ連動設計

* **X軸 (時間軸) の目盛り動的調整**:
  - 横幅ピクセル数を基準として、目盛りが重ならない目標分割数 (`chartWidth / 150`) を算出。
  - ビンサイズ（5秒〜5分）以上の値から、5秒、10秒、30秒、1分、2分、5分、10分、15分、30分、1時間といったキリのいい時間ステップ (`niceIntervalX`) を自動選択し、X軸の最小値・最大値をそのステップで丸めて表示する。
* **Y軸 (コメント数) の目盛り動的調整**:
  - 縦幅ピクセル数を基準として、適切な分割数 (`chartHeight / 50`) を算出。
  - 1目盛りあたりのステップ値 (`targetStep`) から、1, 2, 5, 10, 20, 25, 50, 100, 200, 250, 500 またはそれ以上の10の累乗倍などのキリの良い整数ステップ (`niceStep`) を選択する。
  - グラフの最大表示値をそのステップの倍数に丸め、常に綺麗かつ等間隔な整数目盛りで描画する。
* **リサイズ連動（レスポンシブ描画）**:
  - メインウィンドウのリサイズイベント (`resizeEvent`) をフックし、グラフエリアのサイズが変更されたら即座に再計算 (`updateChartDisplay`) をトリガーする。
* **パフォーマンス保護**:
  - チャット参加者が100人を超える場合、描画処理負荷軽減のために「全員表示」から自動的にコメント数「TOP 10」表示に切り替えるフォールバック機構を適用する。

## 2.7 Twitchチャットアナウンス送信機能の設計

* **UI構成**:
  - コメントタブ（「ビューワ」タブ）の最下部にアナウンス送信用コントロールエリアを追加。
  - アナウンス背景色を選択するプルダウンメニュー (`QComboBox`)、メッセージを入力するテキストボックス (`QLineEdit`)、および送信を実行するボタン (`QPushButton`) を配置する。
* **送信制御**:
  - 送信ボタンクリック、または入力エリアでの Enter キー押下をフック。
  - メッセージが空でない場合にコントローラへ非同期要求を送信。
  - `TwitchEventCollectorImpl` 経由で `POST /helix/chat/announcements` APIを呼び出す。
  - 応答結果（成功または失敗）をメインウィンドウのステータスバー等にフィードバックする。

## 2.8 物理アバターオーバーレイおよびドラッグ同期機能の設計

### 2.8.1 システム構成と通信リレー設計
* **中継サーバー (`ObsWebSocketServer`)**:
  - 管理画面からのドラッグ・ドロップ操作（座標データ）をリアルタイムにOBS側の表示画面へ転送するため、WebSocketサーバーに「メッセージリレー（ブロードキャスト）」機能を設ける。
  - `processTextMessage(const QString& message)` において、受信したテキストデータを、送信元のクライアントソケットを除くすべての接続中ソケットに対して無加工でそのまま送信する。
* **物理オーバーレイ画面 (`overlay_physics.html`)**:
  - Twitchコメント受信イベントおよびWebSocketから中継される座標同期メッセージ（`drag_start`, `drag_move`, `drag_end`）を処理する。
  - `requestAnimationFrame` を用いて、アバターの重力落下、地面（画面下端）での衝突・バウンドの物理シミュレーションを毎フレーム実行する。
  - アバター画像は常に直立状態（回転角0度固定）を維持する。
* **コントロール画面 (`overlay_control.html`)**:
  - 現在出現しているアバターを画面上に並べて表示し、配信主によるドラッグ＆ドロップ操作を受け付ける。
  - ドラッグ開始、移動中、ドラッグ終了の各時点で座標データをWebSocket経由でC++中継サーバーへ送信する。

### 2.8.2 設定パラメータの動的UI表示と共有
* **動的UI制御 (`MainWindow`)**:
  - 設定タブの「オーバーレイファイル」コンボボックス (`comboObsOverlay`) の選択変更シグナルをフックする。
  - 選択されたファイル名が `overlay_physics.html` である場合に限り、設定タブ内の物理オーバーレイ設定用UI（サイズ指定スライダー、バウンド反発係数スライダー、およびサイズプレビュー表示）の可視性 (`setVisible`) を `true` に設定し、それ以外のファイル選択時は `false` にする。
* **設定の永続化とアセット側への即時通知**:
  - 設定パラメータ（ミニマムサイズ、マックスサイズ、バウンド反発係数）は `ConfigManager` を経由して `config.json` に保存する。
  - UI上でパラメータが変更された際、WebSocketサーバー経由ですべてのクライアントに対して設定変更イベント（`settings_changed`）としてパラメータ値を即時ブロードキャストし、表示画面および管理画面へ反映させる。

### 2.8.3 パス制限とセキュリティの設計（ディレクトリトラバーサル・情報漏洩対策）
* **第1の壁：URLパスの文字列フィルタ (`ObsHttpServer`)**:
  - HTTPリクエストを受信した際、ファイルの検索処理に入る前に、パス名（URLデコード前の生リクエスト文字列）を正規表現等の文字列パターンで検査する。
  - 許可文字を「半角英数字、ハイフン(`-`)、アンダースコア(`_`)、スラッシュ(`/`)、ピリオド(`.`)」のみとし、特にスペース（空白）、`%`、`:`、`\`、およびドットの連続（`..`）が含まれている場合は即座にアクセスを拒否（`400 Bad Request` / `404 Not Found`）し、ファイルシステムへの問い合わせを一切行わない。
* **第2の壁：物理的な実絶対パスの範囲検証**:
  - 文字列フィルタを通過したリクエストパスについて、公開フォルダである `assets/overlay/` を起点としたパスを結合し、`QFileInfo::canonicalFilePath()` を用いてシンボリックリンク等を解決した「物理的な実絶対パス」を取得する。
  - この絶対パスの先頭が、ドキュメントルートの絶対パスで始まっていること（`startsWith`）を厳密に検証し、外側のファイル（例: `C:/Program Files/...` や `C:/Users/.../Documents`）へのアクセス要求である場合は、一切のファイル読込を拒否（`403 Forbidden`）する。
* **第3の壁：コンテンツセキュリティポリシー (CSP) の強制適用**:
  - `ObsHttpServer` からHTMLリソース等をブラウザにレスポンス送信する際、レスポンスヘッダーにコンテンツセキュリティポリシー (CSP) を設定して返却する。
  - 送信元・読込元・通信先をローカルホスト（`self`, `localhost`, `127.0.0.1`）およびTwitch公式CDNドメイン（画像配信用）のみに制限し、悪意あるHTMLコードによる外部サーバーへの意図しないデータ送信（情報漏洩）や外部ファイルのロードをブラウザレベルで強制遮断する。

### 2.8.4 高負荷・フリーズ対策（セーフティガバナー）
* **アバター数の上限制限**: 表示画面上の同時物理演算対象アバター数を最大30人に制限し、古いアバターは自動的にフェードアウト消去する。
* **計算負荷（フレーム時間）の監視**: JavaScriptの物理ループ処理において、1フレームの演算処理時間を計測し、10msを超えた場合は異常高負荷と判定して物理シミュレーションを安全に一時停止（サスペンド）させ、ブラウザおよびOBS全体の動作フリーズを防ぐ保護ロジックを実装する。

## 3. テスト戦略 (GTest / QTest)

システムの品質と拡張性を継続的に担保するため、自動テストを標準で組み込む。

### 3.1 テストフレームワーク
* **ロジック・モジュールテスト**: `GTest` (Google Test) / `GMock` (Google Mock)
* **Qt UI・イベント連携テスト**: `QTest`

### 3.2 テストスコープとアプローチ
* **結合テストのスコープ**: 
  UI側のイベントハンドラ（操作の起点）からスタートし、内部ロジック（解析、DB保存等）を通り、最終的に外部モジュールへ「送信命令」が出されるまでのフローを一気通貫でテストする。
* **外部システムのダミー化 (Mocking / 依存性の注入)**:
  * 実際のTwitchサーバーやOBSと通信するとテストが遅延・不安定になるため、外部連携モジュールは**インターフェース（純粋仮想クラス）**を利用して抽象化する。
  * テスト実行時は、これらを **ダミー（Mockオブジェクト）** に差し替える。
  * 【例】: 実際のHTTPリクエストを飛ばす代わりに、`TwitchActionSender` のダミーへ処理を流し、「NGワードを検知した結果、正しい警告文字列がダミーの送信キューに渡されたか」をGTestのアサートで検証する。

これにより、ネットワーク環境に依存せず、CI環境でも瞬時に実行可能な堅牢な自動テスト環境を実現する。
