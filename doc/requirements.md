# Twitchコメント解析ツール 要件定義

## 1. 概要
- 目的: Twitchのコメントをリアルタイムに表示・解析し、BOTとして自動応答やチャット連携を行うツール
- 本ツールは、将来的に開発予定のTwitch配信の総合管理ツールにおいて、コメントの表示、解析、BOT機能を提供するモジュールとして流用されることを前提に設計される。
- 対象: 配信者、モデレーター、チャット参加者
- 基本機能:
  - コメントビューワ
  - コメント解析（感情、キーワード、トレンド）
  - BOT機能（自動応答、コマンド処理、モデレーション）

## 2. ステークホルダー
- **配信者**: 主な利用者。コメントのリアルタイム表示や簡単な操作での配信管理を求める。
- **視聴者**: 配信動画を視聴し、コメントを投稿するユーザー。円滑で楽しいチャット体験を期待。
- **モデレーター**: チャットの秩序を保つために荒らしやスパムを管理する役割を持つ。モデレーション機能を活用。
- **開発者 / 運用担当**: システムの設計、開発、保守を担当し、ツールの安定的な運用を実現する。

## 3. システムの範囲とアーキテクチャ
### 技術スタック
- **開発言語 / フレームワーク**: C++ / Qt6
- **対応OS**: Windows先行対応（設計上は将来的なLinux, Mac等のクロスプラットフォーム対応を考慮する）

### 含む機能とモジュール構成
- 各機能はモジュール化され、必要に応じて個別に有効化・無効化が可能であること。
- コメント取得・表示機能は「標準機能（コア機能）」として提供する。
- 解析やBOT等の拡張機能は「プラグイン」として実装し、あるいはWebSocket等を利用してOBS等の外部ツールに演出として連携可能な設計とする。
- Twitch APIを使ったコメント取得機能
- コメント一覧表示機能
- コメント解析機能（日本語対応）
- BOTとしての自動返信・定型応答機能
- コマンド受付・実行機能
- ログ記録と検索機能
- 設定画面 / 設定ファイルによる挙動制御

### 範囲外
- Twitchの配信映像自体の表示
- 他プラットフォーム（YouTube、Discord等）との連携（初期段階ではTwitchに絞る）

## 4. 機能要件
### 4.1 コメントビューワ
- Twitchチャットおよびイベントをリアルタイムに受信して画面表示
- ユーザー名、コメント内容、タイムスタンプを表示
- ユーザアカウントに設定されたプロフィールアイコンの画像を表示
- コメントフィルタ（キーワード、ユーザー、モデレーション）
- チャンネルポイントの表示および操作（ポイント返却/確定の操作を含む）
- コメント読み上げ機能との連携
  - 棒読みちゃんへのTCPソケット連携
  - テキストファイル書き出し連携
  - VOICEVOX連携
- 重要マークなどの操作

### 4.2 コメント解析
- 日本語コメントの形態素解析
- 感情判定（肯定/否定/中立）
- キーワード抽出
- スパム・荒らし判定
- トレンドワードを検出し通知
- ユーザごとのコメント数を集計し、配信者やモデレーターに提供

### 4.3 BOT機能
- 定型応答
- コマンド応答（例: `!help`, `!rules`, `!time`）
- モデレーション機能
  - NGワード自動削除（自動実行または配信者への処理要求の選択が可能）
  - 違反ユーザー警告（自動実行または配信者への処理要求の選択が可能）
- BOTのON/OFF切り替え

### 4.4 設定・管理
- Twitch接続設定（OAuth、チャンネル指定）
- 解析設定（辞書、閾値、除外ワード）
- BOT設定（コマンド登録、応答テンプレート）

## 5. 外部インターフェース要件
- **Twitch連携 (受信)**: チャットやチャンネルポイント等の各種イベントを統合的・リアルタイムに受信するため「EventSub (WebSocket)」を利用する。
- **Twitch連携 (送信)**: BOT機能としての発言送信などは Twitch API (HTTP) を利用する。

## 6. データ・永続化要件
- **管理方式**: 設定やチャット履歴・集計データの管理には、検索性や集計のしやすさを考慮してローカルDB（SQLite等）を利用する。
- **暗号化・難読化**: プライバシー保護の観点から、DBに保存するチャットの発言者や発言内容は難読化ライブラリ（`TransCipher-Dist` 等）を用いて秘匿化する。
- **保持期間とアーカイブ**: チャットデータの保存期間は1ヶ月とし、7日分ごとにまとめて圧縮・アーカイブ化を行う（圧縮するデータは集計等に必要な情報のみを維持する）。

## 7. 画面・UI/UX要件
- **画面構成**: 1つのメインウィンドウ内に「ビューワ」「解析結果」「設定」等の機能をタブで切り替える方式（タブUI）を採用する。

## 8. 非機能要件
- **パフォーマンス**: 1000コメント/分程度の処理を遅延なく表示
- **信頼性・可用性・エラーハンドリング**:
  - Twitch API切断時の再接続機能
  - セッション切断や再接続成功など、自動復帰が見込めるステータス変化は「画面下部のステータスバー」に表示する。
  - トークン無効、ユーザ設定ミス、アカウントBANなど、ユーザの操作なしには復帰できない問題については「ダイアログ」を表示して警告する。
- **拡張性・モジュール性**: 
  - 将来の他プラットフォーム追加や解析機能追加が容易な設計とする。
  - 将来開発する「総合管理ツール」で再利用可能な設計とする。
- **セキュリティ (OAuthトークン管理)**:
  - OAuthトークンなどの機密情報は平文で保存しない。
  - WindowsのDPAPI機能や、`QtKeychain` 等のOSの資格情報管理機構を利用し、OSのログインユーザのみが復号できる安全な形式で保存する。
- **運用性**: 設定のエクスポート/インポート、ログの参照性

## 9. 制約
- Twitch APIのレート制限に従う
- 日本語チャット向け形態素解析ライブラリを利用

## 10. 受け入れ基準
- Twitchのチャットやイベントが正常に取得・表示できる
- 指定コマンドに対してBOTが正しく応答する
- チャンネルポイントが正しく受信・操作できる
- 解析結果が画面に表示される
- 設定変更が反映される
- 認証情報が暗号化されて安全に保存・読み込みできている

## 11. 次フェーズ
- システム設計: アーキテクチャ、モジュール構成、データフロー
- 詳細設計: 画面レイアウト、API設計、データ構造、DBスキーマ設計
